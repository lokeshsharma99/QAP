The Agentic Judge

The invisible quality gate. A post_hook on every workflow step — never a team member.

🚨
NEVER bypass the Judge. NEVER bypass HumanReviewGateStep. If Judge confidence < 90%, the Human Lead MUST review. No exceptions.

🎯 Trust Logic

confidence ≥ 90%
✅ AUTO-APPROVE — artifact proceeds to next pipeline step
confidence < 90%
⏸ HumanReviewGateStep — artifact held for Human Lead in AgentUI
confidence < 50%
❌ AUTO-REJECT — sent back to producing agent with specific feedback

📋 Judge Variants (DoD Checklists)

VariantReviewsKey Checks
Gherkin Judge Scribe output BA-readable? Reusable steps? Syntax valid? Traceability to Jira AC?
Code Judge Engineer output No hardcoded sleep? Modular POM? eslint passes? Type-check passes? Look-Before-You-Leap followed?
Data Judge Data Agent output PII masked? Unique constraints satisfied? No production data? Cleanup queries present?
Healing Judge Medic output Verification passes 3×? logic_changed == False? Diff is surgical (single locator line)?

📄 JudgeVerdict Contract

class JudgeVerdict(BaseModel):
    artifact_type: str           # "gherkin" | "code" | "data" | "healing"
    agent_id: str               # ID of the producing agent
    confidence: float           # 0.0 – 1.0
    passed: bool                # True if confidence >= 0.90
    checklist_results: dict[str, bool]  # each DoD item pass/fail
    rejection_reasons: list[str]      # empty if passed
    requires_human: bool        # True if confidence < 0.90

🩺 Medic Constraints (Non-Negotiable)

⛔
Medic is ONLY allowed to change locator selectors and wait strategies.
NEVER: assertions, test flow, business logic, method names, imports.
Edits must be surgical — one locator line, not a file rewrite.
Must produce a unified diff for human review.
Verification run must pass 3× consecutively before PR.

📦 How Knowledge Bases are Created

# db/session.py
def create_knowledge(name: str, table_name: str) -> Knowledge:
    return Knowledge(
        name=name,
        vector_db=PgVector(
            db_url=db_url,
            table_name=table_name,
            search_type=SearchType.hybrid,
            embedder=OllamaEmbedder(id="qwen3-embedding:4b", dimensions=2560),
        ),
    )
ℹ️
Git-sync hook: A .git/hooks/post-commit script triggers the Librarian to re-index codebase_vectors on every commit to main/develop. This keeps the Engineer's KB always current.

Agno Features — What They Do & Why We Use Them

Quality Autopilot is built exclusively on Agno (agno[os]). This section explains every Agno capability we use, what it does, and why it matters for the STLC pipeline.

📖
Agno is a pure-Python agent framework. Its primitives are Agent, Team, Workflow, and AgentOS. No LangChain, CrewAI, AutoGen, or LlamaIndex — ever.

🖥️ AgentOS — The Runtime Host

⚡
AgentOS
agno.os.AgentOS · app/main.py

What it is: A FastAPI application host that registers all agents, teams, and workflows under a single process on port 8000. It exposes REST + SSE endpoints, handles session persistence, and manages authorization.

Why we use it: AgentOS turns the entire agent fleet into a single deployable service. The Control Plane talks to it via POST /agents/{id}/runs and Server-Sent Events for streaming. All 16 agents, 6 teams, and 12 workflows are registered once in app/main.py.

agent_os = AgentOS(
    name="Quality Autopilot",
    tracing=True,            # every tool call logged to traces_table
    authorization=RUNTIME_ENV == "prd",
    db=agent_db,
    agents=[architect, scribe, engineer, ...],
    teams=[strategy_team, context_team, ...],
    workflows=[spec_to_code, triage_heal, ...],
)

🤖 Agent — The Core Primitive

🤖
agno.agent.Agent
One agent per directory (agents/<name>/)

What it is: An LLM-powered autonomous unit with its own instructions, tools, memory, and session state. Agents reason, call tools, and produce structured outputs.

Why we use it: Each STLC role (Architect, Scribe, Engineer, etc.) maps to one Agent with a focused system prompt and minimal toolset — single responsibility principle for AI.

🗄️ db=agent_db — Persistent Sessions

What it is: A PostgresDb instance passed to every agent. Agno uses it to persist chat history, session state, memories, and learnings across runs.

Why we use it: Without db, every conversation is stateless and ephemeral. With it, agents remember previous interactions, the Architect can reference a previous RequirementContext, and the Detective can look up past RCA findings — all stored in PostgreSQL, never SQLite.

# db/session.py
agent_db = PostgresDb(id="quality-autopilot-db", db_url=db_url)

# Every agent gets the same db instance
agent = Agent(db=agent_db, ...)

📚 knowledge + search_knowledge — Vector KB Search

What it is: A Knowledge object backed by PgVector. When search_knowledge=True, Agno automatically queries the KB before each response using SearchType.hybrid (vector similarity + keyword BM25 combined).

Why we use it: The Engineer queries codebase_vectors to check if a Page Object already exists before writing new code. The Medic queries rca_kb for historical fix patterns. The Architect queries qap_learnings for conventions. This prevents hallucinated selectors and duplicate code.

# db/session.py — all KBs use hybrid search + Ollama embedder
automation_kb = Knowledge(
    vector_db=PgVector(
        table_name="codebase_vectors",
        search_type=SearchType.hybrid,
        embedder=OllamaEmbedder(id="qwen3-embedding:4b", dimensions=2560),
    ),
)
engineer = Agent(
    knowledge=automation_kb,
    search_knowledge=True,  # auto-searches KB on every response
    ...
)

🎓 learning + add_learnings_to_context — Agents Learn

What it is: learning=True gives agents tools to record learnings into the qap_learnings KB. add_learnings_to_context=True automatically prepends relevant past learnings to every new response.

Why we use it: Over time, agents accumulate project-specific knowledge — stable locator patterns, quirky API behaviours, known flaky test areas. This is injected automatically as context, making each run smarter than the last without manual prompt engineering.

detective = Agent(
    learning=True,                    # can store learnings
    add_learnings_to_context=True,   # past learnings injected as context
    ...
)

🧠 enable_agentic_memory — Per-User Memory

What it is: Agno provides agents with tools to store and retrieve facts about the user across sessions. Memory is user-scoped and persisted in PostgreSQL — it survives container restarts.

Why we use it: The Concierge remembers your preferred AUT URL. The Architect remembers that you always want P0 tickets addressed first. The Scribe remembers your Gherkin step naming conventions. These preferences are recalled automatically — the agent adapts to you over time, not just within a single chat.

Control Plane: View, search, and delete all memories at /memory. The "Optimize" button triggers memory consolidation to deduplicate and summarise stale entries.

architect = Agent(
    enable_agentic_memory=True,  # user facts persisted across sessions
    ...
)

📋 session_state + enable_agentic_state — Run-Time State

What it is: session_state is a dictionary of transient data that persists across turns within a session (not across sessions like memory). enable_agentic_state=True gives the agent tools to read and modify it. add_session_state_to_context=True injects it as context on every turn.

Why we use it: The Engineer tracks created_files, current_feature, and validation_results across a multi-turn automation generation session. The Detective tracks analyzed_failures and root_causes. This prevents the agent from losing track of its progress mid-task.

engineer = Agent(
    session_state={
        "created_files": [],
        "current_feature": None,
        "validation_results": {},
    },
    enable_agentic_state=True,
    add_session_state_to_context=True,
    ...
)

💬 add_history_to_context + read_chat_history — Multi-Turn Awareness

What it is: add_history_to_context=True automatically prepends the last N turns of the conversation. read_chat_history=True gives the agent a tool to retrieve further history on demand. num_history_runs=5 sets the default window.

Why we use it: Without history, every message is treated as a new conversation — the Scribe forgets which feature it was writing, the Medic forgets which locator it was patching. With 5-turn history, agents maintain coherent multi-step task execution.

medic = Agent(
    add_history_to_context=True,
    read_chat_history=True,
    num_history_runs=5,
    ...
)

🗜️ CompressionManager + STLC_COMPRESSION_PROMPT — Token Budget

What it is: Agno's CompressionManager compresses long tool call histories when they exceed a token threshold. The compress_tool_call_instructions parameter tells the LLM how to compress — what to preserve vs. drop.

Why we use it: Generic compression (Agno default) loses STLC-critical details like selector values, RCA classifications (LOCATOR_STALE), Playwright error messages, and ticket IDs. Our custom STLC_COMPRESSION_PROMPT in app/settings.py instructs the compressor to always preserve these domain-specific signals — keeping long triage or code-gen sessions coherent without blowing the context window.

Applied to: All 16 agents and 6 teams (globally via app/settings.py).

# app/settings.py
STLC_COMPRESSION_PROMPT = """
You are compressing a tool-call history for a software testing agent.
ALWAYS preserve:
- Playwright locator selectors and data-testid values
- RCA classifications: LOCATOR_STALE, DATA_MISMATCH, TIMING_FLAKE, ENV_FAILURE, LOGIC_CHANGE
- Playwright error messages and stack traces
- Ticket IDs (Jira/ADO) and acceptance criteria IDs
- Feature file paths and POM class names
- Judge confidence scores and verdict outcomes
Drop: repeated intermediate reasoning, verbose OK/200 responses, duplicate tool echoes.
"""

# Every agent/team uses this pattern
agent = Agent(
    compression_manager=CompressionManager(
        compress_tool_call_instructions=STLC_COMPRESSION_PROMPT,
    ),
    ...
)

🔧 tools=[] — Capabilities

What it is: A list of callables or Agno tool classes that the agent can invoke during its reasoning loop. Each tool is a function with typed parameters and a docstring — Agno automatically generates the JSON schema for the LLM.

Why we use it: Tools are what make agents take real actions — reading files, searching KBs, running code, calling APIs, writing to GitHub. Without tools, agents can only reason in text.

👥 Team + TeamMode.coordinate — Agent Collaboration

What it is: agno.team.Team is a group of agents with a shared leader. TeamMode.coordinate means the team leader decides which member to delegate to — members never delegate to each other directly.

Why we use it: The Strategy Squad leader receives "Generate tests for GDS-123" and decides to first send it to Architect, then pass the RequirementContext to Scribe. This coordination happens automatically without manual routing. share_member_interactions=True means all members see each other's work — so Scribe can see the full Architect output when generating Gherkin.

strategy_team = Team(
    mode=TeamMode.coordinate,    # leader delegates to members
    members=[architect, scribe],
    share_member_interactions=True, # members see each other's work
    show_members_responses=True,  # responses streamed to UI
    ...
)

🔀 Workflow Primitives — Step, Condition, Router, Loop

▶️
Step
agno.workflow.Step

A single agent invocation within a pipeline. Passes the output of the previous step as input. Most workflow nodes are Steps.

🔀
Condition
agno.workflow.Condition

A gate that only runs its child steps if the evaluator function returns True. Used for all Judge gates — pipeline only continues if confidence ≥ 90%.

🛣️
Router
agno.workflow.Router

Selects one branch from multiple options based on the evaluator output. Used in the triage pipeline to route LOCATOR_STALE to Medic vs. other classifications to Human.

🔁
Loop
agno.workflow.Loop

Repeats its child steps until a stop condition is met. The Triage & Heal pipeline uses a Loop to verify the Medic's patch passes 3 consecutive times before submitting the PR.

⚖️ post_hook — The Invisible Judge Pattern

What it is: A post_hook is a function called after every agent response within a workflow step. Agno passes the full step output to it.

Why we use it: The Judge is wired as a post_hook on every workflow Step — not as a team member. This makes it invisible to other agents (they can't delegate to it or skip it), and it runs automatically on every artifact without requiring any agent to "remember" to call it. This is how we guarantee that no unreviewed artifact ever leaves a pipeline step.

# Judge as a post_hook on a workflow Step
def judge_post_hook(response):
    verdict = judge.run(f"Review this artifact:\n{response.content}")
    confidence = extract_confidence(verdict.content)
    if confidence < 0.50:
        raise AutoRejectError(verdict.content)
    elif confidence < 0.90:
        queue_for_human_review(verdict)

Step(name="Author Gherkin", agent=scribe, post_hooks=[judge_post_hook])

📝 markdown=True — Structured Output

What it is: Instructs Agno to tell the LLM to format responses as Markdown. The Control Plane renders markdown, code blocks, and Mermaid diagrams.

Why we use it: All agents use markdown=True. This gives consistent rich output in the Chat UI — code blocks with syntax highlighting, structured tables, numbered steps — making agent responses readable and reviewable by human leads without parsing raw text.
